# Design Spec: Volt Personal Knowledge OS (Planning, Automation, and Notifications)

## Goal Description
Transform Volt from a bookmark and note manager into a cohesive **Personal Knowledge Operating System (PKOS)**. This redesign establishes clear boundaries between **Knowledge** (Resources, Notes, Projects, People), **Planning** (Reminders, Watchlist, Calendar Feed), **Automation** (Registry-Based Scheduler, Metadata Sync, Notification Engine), and **Experience** (Dashboard widgets, Command Palette, Notification Popover). 

---

## Architecture Blueprint

```text
Presentation (Experience)
├── Dashboard (Inbox, My Day, Upcoming, Continue Working)
├── Command Palette (Unified Search: Notes, Projects, Resources, Reminders, Watchlist)
├── Header Notifications (Popover Bell with GSAP micro-animations)
└── Calendar Feed (Standard ICS subscription)

Knowledge
├── Resources (Websites, YouTube, GitHub, etc.)
├── Notes (Rich Knowledge capture)
├── Projects (Workspace organization)
├── People (Collaborators, contact logs)
└── Watchlist (Movies, TV Series, Anime)

Planning
├── Reminders (Lightweight, status-driven, multi-attachment tasks)
├── Watchlist States (Upcoming, Released, Watching, Completed, Dropped, Paused)
└── Digests (User-configurable Morning & Weekly summaries)

Automation
├── Scheduler (Cron-driven modular job runner)
├── Job Registry (Metric tracking: execution logs, failure counts)
├── Metadata Sync Service (TMDb / AniList background updates)
└── NLP Date Parser (Chrono-Node local parse with LLM fallback)
```

## Proposed Database Schemas

### 1. [NEW] Reminders Collection (`reminders`)
Reminders are lightweight action items distinct from static notes.

```typescript
export type ReminderPriority = "P1" | "P2" | "P3" | "P4"; // Critical, High, Medium, Low
export type ReminderStatus = "pending" | "completed" | "cancelled" | "expired";

export interface ReminderAttachment {
  type: "note" | "project" | "person" | "resource" | "watchlist";
  id: string; // Target document string ID
}

export interface Reminder {
  _id?: string | ObjectId;
  userId: string;
  title: string;
  description?: string;
  status: ReminderStatus;
  priority: ReminderPriority;
  triggerAt: Date;
  attachments: ReminderAttachment[]; // Supported multiple links
  
  notification: {
    lastNotifiedAt?: Date;
    sentAt?: Date;
  };
  
  recurrence?: {
    frequency: "daily" | "weekly" | "monthly" | "yearly" | "none";
    interval?: number; // e.g., every 2 weeks
  };
  
  createdAt: Date;
  updatedAt: Date;
}
```

### 2. [NEW] Notifications Collection (`notifications`)
Notifications are transient timeline events generated by the planning and automation layers.

```typescript
export const NotificationTypes = {
  WATCHLIST_RELEASE: "watchlist.release",
  WATCHLIST_EPISODE: "watchlist.episode",
  REMINDER_DUE: "reminder.due",
  PROJECT_DEADLINE: "project.deadline",
  SYSTEM: "system",
} as const;

export type NotificationType = typeof NotificationTypes[keyof typeof NotificationTypes];

export interface Notification {
  _id?: string | ObjectId;
  userId: string;
  title: string;
  message: string;
  type: NotificationType;
  link?: string; // Target link for click-to-navigate, e.g., "/media-watchlist?id=123"
  createdAt: Date;
  readAt?: Date;
  dismissedAt?: Date;
  archivedAt?: Date;
}
```

### 3. [NEW] Job Execution Metrics Collection (`job_metrics`)
Tracks scheduler runs to verify system health.

```typescript
export interface JobMetric {
  _id?: string | ObjectId;
  jobName: string;
  startedAt: Date;
  finishedAt: Date;
  durationMs: number;
  success: boolean;
  itemsProcessed: number;
  error?: string;
}
```

## Proposed Changes & Components

### 1. Automation Layer: Unified Job Registry Scheduler
#### [NEW] [route.ts](file:///c:/Users/ADMIN/Desktop/shammas/website/volt/app/api/cron/scheduler/route.ts)
Exposes a single Vercel Cron-triggered HTTP GET endpoint.
*   Authenticates via `Authorization: Bearer ${CRON_SECRET}`.
*   Uses a **Job Registry Pattern**:
    ```typescript
    interface Job {
      name: string;
      run(): Promise<{ itemsProcessed: number }>;
    }
    ```
*   Iterates through registered jobs (`ReminderJob`, `WatchlistSyncJob`, `DigestJob`, `CleanupJob`).
*   Saves execution statistics to the `job_metrics` collection.
*   Enforces a timeout-safe sync limit (e.g., maximum 10 items processed per metadata sync run).

#### [NEW] [jobs.ts](file:///c:/Users/ADMIN/Desktop/shammas/website/volt/lib/jobs/jobs.ts)
Contains implementations of the registered jobs:
*   **ReminderJob**: Queries `reminders` with status `pending`, `triggerAt <= now`. Generates event-based notification, marks status `pending` (or updates to `completed` if single-use, or schedules next recurrence).
*   **WatchlistSyncJob**: Selects watchlist items needing sync (oldest `sync.lastChecked`), fetches details from TMDb/AniList, updates release dates and posters, and triggers release notifications.
*   **CleanupJob**: Triggers hard-deletion of notifications where `createdAt <= now - 90 days`.

### 7. Planning Layer: Natural Language Quick Capture
#### [NEW] [parser.ts](file:///c:/Users/ADMIN/Desktop/shammas/website/volt/lib/utils/parser.ts)
A tiered parsing engine executed during reminder capture:
1.  **Tier 1 (Chrono-Node)**: Attempt local extraction using the `chrono-node` library. It resolves 90% of expressions (e.g. "Submit invoice tomorrow 3pm") with zero latency and zero cost.
2.  **Tier 2 (AI Fallback)**: If local Chrono-Node parsing fails to identify a valid date/time, trigger a quick Next.js Server Action invoking the Google Gemini API with a structured JSON schema prompt to extract the intent.
3.  **Tier 3 (Manual Picker)**: If both automated systems fail, fallback gracefully to a standard date-time selector.

### 8. Experience Layer: Calendar Subscription (ICS)
#### [NEW] [route.ts](file:///c:/Users/ADMIN/Desktop/shammas/website/volt/app/api/users/calendar/route.ts)
*   Generates a standard RFC 5545 compliance iCalendar (ICS) feed.
*   Accepts secure token queries: `/api/users/calendar?token=xyz`.
*   Assembles a chronological list of:
    *   Pending & completed reminders (`triggerAt` mapped to calendar time events).
    *   Watchlist items with digital/theatrical release dates (mapped to all-day events).
    *   Airing TV and Anime episodes (mapped to specific airing timestamps).

## Verification Plan

*   **Unit Tests**: Run parsing test suite targeting `chrono-node` logic to verify correct resolution of terms like "next Tuesday at 9am".
*   **Cron Execution Simulation**: Local trigger tests of `/api/cron/scheduler` checking job run times and verified creation of entries in `job_metrics`.
*   **ICS Feed Validator**: Pass generated ICS outputs through standard CalDAV/iCalendar format check engines to verify feed parsing.

### Manual Verification
*   Validate dashboard widget updates on completing reminders.
*   Test command palette filtering for new reminder elements.
*   Verify that notification bell popover works correctly on mobile viewport scaling.
